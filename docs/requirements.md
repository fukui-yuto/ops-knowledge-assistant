# 要件定義書

## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| プロジェクト名 | ops-knowledge-assistant |
| 目的 | 運用手順書の自動生成による作成工数削減・品質均一化 |
| 対象ユーザー | インフラ運用チーム（SRE / 情シス） |
| 想定規模 | 手順書 100〜500件、チケット 1,000〜10,000件 |

## 2. 機能要件

### 2.1 ナレッジ取り込み (Ingestion)

| ID | 要件 | 優先度 |
|---|---|---|
| F-ING-001 | Markdownファイルを単体で取り込める | Must |
| F-ING-002 | ディレクトリ一括取り込みができる | Should |
| F-ING-003 | 取り込み時にcontent_hashで差分検知し、未変更ファイルはスキップする | Must |
| F-ING-004 | source_type (procedure / ticket / config / log) で分類できる | Must |
| F-ING-005 | source_system (confluence / jira / proxmox 等) で分類できる | Must |
| F-ING-006 | チケットには severity / status / affected_system 等の構造化フィールドを保持できる | Must |
| F-ING-007 | 取り込み履歴を ingestion_log に記録する | Must |
| F-ING-008 | JIRA API からチケットを自動同期できる | Could |
| F-ING-009 | Confluence API から手順書を自動同期できる | Could |
| F-ING-010 | 既存ドキュメント更新時、旧チャンク・ベクトルを自動削除して再取り込みする | Must |
| F-ING-011 | ディレクトリ監視により、ファイル追加・更新・削除を検知し自動で取り込み/削除する | Must |
| F-ING-012 | ドキュメント削除時、PostgreSQL・ChromaDB・LocalStorageの全データを自動で連動削除する | Must |
| F-ING-013 | 同期状態の整合性チェックコマンドを提供する（DB・ChromaDB・ファイルの不整合検出） | Should |

### 2.2 テンプレート管理

| ID | 要件 | 優先度 |
|---|---|---|
| F-TPL-001 | テンプレートはMarkdownファイルとして `data/templates/` に配置する | Must |
| F-TPL-002 | テンプレートにプレースホルダ（`{{title}}`, `{{system}}` 等）を使用できる | Must |
| F-TPL-003 | 複数テンプレートから用途に応じて選択できる | Must |
| F-TPL-004 | テンプレート一覧をCLIで確認できる | Must |
| F-TPL-005 | テンプレートにカテゴリタグを付与できる（例: network, storage, k8s） | Should |
| F-TPL-006 | テンプレートのバージョン管理はGitで行う | Must |

### 2.3 手順書生成 (Generation)

| ID | 要件 | 優先度 |
|---|---|---|
| F-GEN-001 | タイトルと説明文を入力すると新規手順書をMarkdownで生成する | Must |
| F-GEN-002 | テンプレートの構成（見出し・セクション順）に従って生成する | Must |
| F-GEN-003 | ベクトル検索で関連する過去手順を自動取得し、参考にして生成する | Must |
| F-GEN-004 | 追加コンテキスト（対象サーバー名、制約条件等）を渡せる | Must |
| F-GEN-005 | 使用するテンプレートを指定できる | Must |
| F-GEN-006 | 参照する過去手順の最大数を指定できる | Should |
| F-GEN-007 | 生成結果をファイル出力またはstdout出力できる | Must |
| F-GEN-008 | 生成履歴（誰が・いつ・何を生成したか）をDBに記録する | Should |
| F-GEN-009 | 生成された手順書に「TODO: 要確認」を含む場合に警告を出す | Should |
| F-GEN-010 | 生成結果をそのまま取り込み（Ingest）してナレッジに追加できる | Could |

### 2.4 検索 (Retrieval)

| ID | 要件 | 優先度 |
|---|---|---|
| F-RET-001 | 自然言語クエリでベクトル類似検索ができる | Must |
| F-RET-002 | source_type / source_system でフィルタリングできる | Must |
| F-RET-003 | チャンク単位の検索結果からドキュメント全文を取得できる | Must |
| F-RET-004 | ハイブリッド検索（ベクトル + キーワード）ができる | Could |
| F-RET-005 | 検索結果のリランキングができる | Could |

## 3. 非機能要件

### 3.1 性能

| ID | 要件 | 目標値 |
|---|---|---|
| NF-PERF-001 | 手順書生成のレスポンス時間 | 30秒以内 |
| NF-PERF-002 | ベクトル検索のレスポンス時間 | 2秒以内 |
| NF-PERF-003 | 単一ファイル取り込み時間 | 10秒以内 |
| NF-PERF-004 | 同時利用ユーザー数 | 5人（初期） |

### 3.2 信頼性

| ID | 要件 |
|---|---|
| NF-REL-001 | ChromaDBが破損してもPostgreSQLのchunksから再構築できる |
| NF-REL-002 | LLM API障害時にわかりやすいエラーメッセージを返す |
| NF-REL-003 | 取り込み途中で失敗した場合、部分的なデータが残らない（トランザクション整合性） |

### 3.3 セキュリティ

| ID | 要件 |
|---|---|
| NF-SEC-001 | APIキーは環境変数で管理し、コードやログに含めない |
| NF-SEC-002 | 生成した手順書に機密情報（パスワード等）を含めない指示をプロンプトに含める |
| NF-SEC-003 | DB接続情報は環境変数で管理する |

### 3.4 運用

| ID | 要件 |
|---|---|
| NF-OPS-001 | Docker Compose で全コンポーネントを起動できる |
| NF-OPS-002 | ログはJSON形式で構造化出力する |
| NF-OPS-003 | ヘルスチェックエンドポイントを提供する（Web API化時） |

### 3.5 拡張性

| ID | 要件 |
|---|---|
| NF-EXT-001 | LLMモデルを設定変更のみで切り替えられる（Gemini → OpenAI等） |
| NF-EXT-002 | ストレージバックエンドを差し替えられる（Local → MinIO/S3） |
| NF-EXT-003 | Web UI / Slack Bot 等のフロントエンドを後から追加できる構造にする |

## 4. 制約事項

| 項目 | 制約 |
|---|---|
| LLM | Google Gemini API（初期）。APIキーに利用上限あり |
| Embedding | text-embedding-004。768次元。バッチ上限あり |
| DB | PostgreSQL 16+。ローカル or Docker |
| ベクトルDB | ChromaDB（組み込みモード）。大規模時はクライアント/サーバーモードに移行 |
| ファイル形式 | 入力・出力ともMarkdown (.md) |
| 言語 | 手順書は日本語 |

## 5. 用語定義

| 用語 | 定義 |
|---|---|
| テンプレート | 手順書の雛形。セクション構成と記載ルールを定義するMarkdownファイル |
| 過去手順 | 既存の運用手順書。ナレッジベースとして取り込み済みのもの |
| チャンク | ドキュメントを分割した単位。ベクトル検索・LLMコンテキストの基本単位 |
| source_type | ドキュメント種別: procedure / ticket / config / log |
| source_system | ドキュメントの出元システム: confluence / jira / proxmox / k8s 等 |
| Ingestion | ドキュメントの取り込み処理（保存→チャンク→Embedding→DB登録） |
| Generation | テンプレ + 過去手順 + 指示 → LLMで新規手順書を生成する処理 |

## 6. ユースケース

### UC-001: 新規手順書の作成
1. ユーザーがタイトルと説明を入力する
2. システムがテンプレートをロードする
3. システムが関連する過去手順をベクトル検索で取得する
4. システムがLLMに（テンプレート + 過去手順 + ユーザー指示）を渡す
5. LLMが新規手順書を生成する
6. ユーザーがMarkdownファイルとして受け取る

### UC-002: 過去手順の取り込み
1. ユーザーがMarkdownファイルとメタデータ（種別・システム・ID・タイトル）を指定する
2. システムがファイルを保存し、チャンク分割・Embeddingを行う
3. PostgreSQLとChromaDBに登録する

### UC-003: テンプレートの追加
1. ユーザーが `data/templates/` にMarkdownファイルを配置する
2. CLI で `--list-templates` で確認する
3. 生成時に `--template <name>` で指定する

### UC-004: 既存手順書の更新
1. ユーザーが同じ external_id で更新版ファイルを取り込む
2. システムが content_hash を比較し変更を検知する
3. 旧チャンク・ベクトルを削除し、新しい内容で再登録する
